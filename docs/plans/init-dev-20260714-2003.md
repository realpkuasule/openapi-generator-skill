## 1. 初始化范围与假设

首版只交付需求规格中的基础闭环：

1. 一份同时供 Codex 和 Claude Code 使用的核心 Skill；
2. 强制“只读识别 → 多轮问答 → 工作边界摘要 → 批准 → 执行”的行为契约；
3. 三个无外部副作用的确定性辅助能力：项目识别、治理档案校验、生成目录比较；
4. 工具中立的决策与全生命周期参考资料；
5. Animator、Revoice、范围扩张三个 eval 场景；
6. 可复现的结构、契约、单元和烟雾验证。

仓库不会为此启动常驻 HTTP 服务。`contracts/openapi-engineering.openapi.yaml` 使用 OpenAPI 3.1 描述平台无关的控制面能力及 JSON 输入输出；首版 CLI 脚本实现相同语义，未来 MCP/HTTP adapter 必须复用该契约。治理档案由独立 JSON Schema 持有权威定义，OpenAPI 通过外部 `$ref` 引用，避免双重权威。

行号说明：新建文件以 `L1` 起算；下列范围是实施锚点。每个阶段完成后回写为实际行号，任何超出范围的新增能力必须先更新本计划和 OpenAPI 契约。

## 2. TDD 与 Contract-First 执行纪律

每项功能严格按以下顺序：

1. **Contract**：先更新 OpenAPI/JSON Schema 与示例；
2. **RED**：先写测试并运行，确认失败原因是能力尚未实现；
3. **GREEN**：实现使测试通过的最小代码；
4. **REFACTOR**：只消除本次引入的重复，保持测试全绿；
5. **VERIFY**：运行阶段门禁并记录实际结果。

禁止：先写实现后补测试、使用 `latest`、自动安装依赖、为未来能力创建空抽象、在测试中访问网络或真实项目写路径。
