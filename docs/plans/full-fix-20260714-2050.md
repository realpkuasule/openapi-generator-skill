## 0. 目标、边界与执行原则

### 0.1 修复目标

本计划把当前“结构正确、静态测试为绿”的初始化原型，推进到可重复、可审计、可自动判定的双平台 Skill。完成后应具备：

1. 治理档案只有一份结构权威，校验器不再复制字段和枚举；
2. OpenAPI、JSON Schema、示例和 CLI 输出通过真实 Schema 校验与语义不变量校验；
3. 项目识别不会被 worktree、生成目录或缓存污染，并能报告未知、冲突和截断；
4. 治理档案能够复用、检测冲突、记录工具状态、feature gap、spike、digest 和实际门禁结果；
5. Eval 不再只是手写答案，而能在隔离环境执行、评分、保存 transcript 和证明批准前零写入；
6. Codex 与 Claude Code 对同一输入进行真实 forward test，并按语义等价而非逐字一致评分；
7. Assess、Audit、Upgrade 和 empirical spike 至少具有确定性、默认无写入的自动化执行器；
8. 一个跨平台 Python 入口汇总所有门禁，缺工具或未运行时返回 `blocked/unverified`，不得伪装为通过。

### 0.2 非目标

- 在 Checkpoint C 批准前不下载、升级或调用 OpenAPI Generator；批准后只允许在隔离临时目录执行固定版本实证门；
- 不修改 Animator、Revoice 或其他目标项目；
- 不自动访问厂商 API、凭据、付费服务或生产环境；
- 不创建常驻 HTTP/MCP 服务；
- 不把生成 DTO 提升为领域模型真源；
- 不把 Codex/Claude Code 的输出逐字一致当作等价性标准；
- 不在没有新边界批准时执行全局 Skill 安装、真实 Agent 会话或依赖下载。

### 0.3 三个批准检查点

| 检查点 | 首次出现 | 需要批准的副作用 | 未批准时允许的动作 |
|---|---|---|---|
| A：本仓库修复 | P0 开始前 | 修改本仓库契约、测试、脚本和锁文件；在隔离开发环境下载锁定的验证依赖 | 只读审计、计划完善 |
| B：真实双平台 Eval | P1-5 开始前 | 调用 Codex/Claude Code CLI，可能产生模型用量或网络流量 | 合成 fixture 和本地 scorer 测试 |
| C：生成器实证门 | P2-1 开始前 | 下载/运行固定版本生成器及目标语言工具链 | dry-run、命令计划和环境诊断 |

所有检查点都必须先给出完整工作边界摘要。批准 A 不自动批准 B 或 C。

### 0.4 TDD 证据规则

每个任务严格执行：

1. 先改权威契约和示例；
2. 添加针对缺口的最小失败测试；
3. 运行并记录预期 RED，失败必须来自目标能力缺失；
4. 实现最小变更；
5. 运行并记录 GREEN；
6. 运行阶段门禁并生成机器可读报告。

RED/GREEN 证据统一写入：

```text
docs/verifications/full-fix-20260714-2050/
├── manifest.json
├── P0-01-red.log
├── P0-01-green.log
├── ...
├── deterministic-report.json
├── forward-eval-report.json
└── final-report.json
```

P2 继续使用同一规则，证据落在 `docs/verifications/p2-20260715/`，并由该目录的 `manifest.json` 统一登记摘要。

`manifest.json` 记录任务 ID、命令、开始/结束时间、退出码、输入文件 digest 和日志 digest。测试未实际运行时不得手工创建通过记录。

## 2. 契约变更（必须先于实现）

修改 `contracts/schemas/governance-profile.schema.json`：

- `rejected_options` 改为 `{option, reason}` 对象数组，与决策 reference 一致；

## 3. 自动化架构

```text
契约/Schema
  ↓
契约测试 RED
  ↓
确定性实现与单元测试
  ↓
scripts/verify.py --tier deterministic
  ↓
隔离 Skill Eval + 文件哈希门
  ↓
Codex / Claude Code adapter forward test
  ↓
语义等价评分 + 机器报告
  ↓
获批后 empirical generator/upgrade gate
  ↓
最终验收报告与 Profile evidence
```

### 3.1 统一验证入口

计划新增：

```bash
uv sync --locked
uv run python scripts/verify.py --tier deterministic --report <path>
uv run python scripts/verify.py --tier forward --adapter codex --adapter claude --report <path>
uv run python scripts/verify.py --tier full --report <path>
```

退出语义：

- `0`：请求的全部门禁实际通过；
- `1`：发现确定性失败或行为不符合；
- `2`：环境、授权或依赖阻断；报告必须是 `blocked/unverified`；
- 任何子门禁缺失都不能通过跳过转换为 `0`。

### 3.2 Eval 隔离原则

- compact fixtures 在临时目录运行，避免测试真实项目写路径；
- 真实 Animator/Revoice forward test 默认只读，并比较运行前后文件树、内容、依赖清单和 Agent 配置哈希；
- runner 只把 prompt、Skill 路径和项目原始证据给被测 Agent，不泄露 `expected`；
- scorer 在 Agent 结束后读取 expected，按模式、问题簇、边界、等待点、决策和副作用评分；
- transcript 和结果自动脱敏；原始凭据、环境变量值和敏感 payload 不进入报告。

## 8. 风险与回退

| 风险 | 控制 | 回退 |
|---|---|---|
| Profile v1 尚未发布但可能已有私下使用者 | 执行前扫描已存在 Profile；发现消费者则改为 v2+迁移 | 保留旧 Schema/validator fixture，停止原地升级 |
| 标准 validator 引入依赖 | 锁版本、冷启动 CI、doctor 输出 | JSON-only 基础诊断仍可运行，但正式 gate 标记 blocked |
| 真实 Agent Eval 产生费用/网络或资源压力 | 检查点 B、case/次数/timeout 上限、单平台串行、每样本原子检查点 | 清理整个 CLI 进程组，保留可续跑 blocked report，不改项目；只重试非通过槽位 |
| Eval 受模型波动影响 | hard gates + 多次采样 + 语义评分 | 不降低 hard gate；记录波动并重新评估 prompt/Skill |
| 真实项目包含敏感数据 | 默认 compact fixture；真实项目只读、最小上下文、报告脱敏 | 删除临时 sandbox/transcript，保留脱敏摘要 |
| 安装器造成双平台分叉 | dry-run、digest、拒绝覆盖 | 只删除本次创建的链接/副本，恢复先前路径 |
| 当前仓库不是 Git 仓库 | 实施前在仓库外创建临时完整快照和 manifest | 从快照恢复本计划涉及文件；不擅自 `git init` |
| 生成器或工具链不可用 | capability probe，exit 2 | 保持现状并报告 unverified，不宣称完成 |
